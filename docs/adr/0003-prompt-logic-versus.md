# ADR-0003: Prompt vs Logic 역할 분리

## 1. 배경 (Context)
Agent(AI)와 협업하여 코드를 작성하고 시스템을 구축하는 과정에서, "Agent에게 지시할 내용(Prompt)"과 "시스템이 실제로 구현해야 할 구조 및 명세(Logic)"가 혼재될 경우 여러 문제점이 발생했습니다. 컨텍스트가 복잡해지면 Agent가 잘못된 구현을 할 위험이 커지며, 향후 프로젝트의 히스토리와 실제 의도를 파악하기 어려워집니다.

## 2. 결정 (Decision)
프로젝트 문서 관리 체계에서 **Prompt(명령어)**와 **Logic(로직/명세)**의 역할을 명확히 정의하고, 각각 다른 폴더 및 파일 명명 규칙으로 물리적인 분리를 강제하기로 결정했습니다.

## 3. 이유 (Reasoning)
- **관심사의 분리**:
  - **Prompt**: 사용자가 Agent에게 내린 실제 지시문이나 대화의 맥락입니다. 이는 시스템이 과거에 '어떤 지시'를 통해 현재 상태가 되었는지를 기록하는 히스토리 역할을 합니다. (`docs/prompt/` 폴더 내에 저장)
  - **Logic**: Agent가 사용자 명령을 수행하기 위해 반드시 준수해야 하는 일련의 규칙, 아키텍처 요구사항, 시스템 명세입니다. (`docs/` 폴더 바로 아래 `phase*-*.md` 형태로 저장)
- **Agent의 인지 부하 감소**: 이 둘을 분리함으로써 Agent는 지시를 수행할 때 명세서(Logic)를 읽고 흔들림 없이 구현에만 집중할 수 있습니다.

## 4. 결과 (Consequences)
- 문서의 유지보수성이 대폭 향상됩니다.
- 추후 프로젝트를 인수인계하거나 새로운 Agent 세션 환경에서도 명세서(Logic)와 ADR 문서만으로 프로젝트의 의도와 동작 원리를 완벽히 파악시킬 수 있습니다.
- 새로운 기능을 구현할 때, 새롭게 작성해야 하는 '지시 사항(Prompt)'과 업데이트해야 할 '명세서(Logic)'의 구분이 명확해집니다.
