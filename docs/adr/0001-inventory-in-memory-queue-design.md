# ADR-0001: Inventory Controller의 인메모리 큐 설계 결정

## 1. 배경 (Context)
Inventory Controller는 클러스터의 리소스 변경을 감시하고 이를 Git에 백업한다. Git 연산(Commit, Push)은 네트워크 지연과 저장소 락(Lock) 이슈로 인해 Reconcile 루프 내에서 동기(Synchronous) 방식으로 처리할 경우 전체 컨트롤러의 성능 저하와 병목을 유발할 수 있다. 따라서 리소스 변경 감지와 실제 Git 반영을 분리하기 위한 중간 버퍼(Queue)가 필요하다.

## 2. 결정 (Decision)
Redis나 RabbitMQ와 같은 외부 메시지 브로커를 사용하지 않고, **Go 언어의 채널(Channel) 및 `client-go`의 인메모리 작업 큐(WorkQueue)를 활용한 비동기 배치 워커**를 구현하기로 결정한다.

## 3. 이유 (Reasoning)
- **장애 복구력 (Self-healing):** 쿠버네티스 오퍼레이터는 재시작 시 전체 리소스를 다시 List/Watch 한다. Pod가 죽어서 메모리 큐가 날아가더라도, 재시작 후 해시 비교를 통해 Git과 다른 상태를 자동으로 감지하고 다시 큐에 쌓으므로 데이터 유실 위험이 없다.
- **운영 복잡도 감소:** 외부 브로커를 설치하고 관리할 인프라 비용과 운영 공수를 절감할 수 있다.
- **충분한 성능:** 관리 대상 리소스(Argo CD Apps, Secrets 등)의 규모가 API 서버와 인메모리 큐가 처리하지 못할 정도로 거대하지 않으며, `client-go`의 Informer 패턴을 통해 이미 최적화된 로컬 캐시를 사용한다.

## 4. 결과 (Consequences)
- 컨트롤러 내부에 배치 워커(Goroutine) 로직이 추가되어야 한다.
- Pod 재시작 직후에는 전체 리소스를 다시 확인하는 과정에서 일시적인 CPU/네트워크 트래픽이 발생할 수 있으나, 쿠버네티스 표준 동작 범위 내이다.

## 5. 구현

### 5.1. 큐 위임 인터페이스 구조
```go
// internal/inventory/reconciler.go
type ReconcileDeps struct {
	Client         client.Client
	Cache          *HashCache
	GitQueue       *git.BatchWorker // 비동기 배치 워커 (인메모리 큐 역할)
	ExportPathTmpl string
}

func (d *ReconcileDeps) handle(ctx context.Context, req ctrl.Request, obj client.Object, kind string) (ctrl.Result, error) {
    // (중략) 메타데이터 정규화 및 해시 비교 검증 로직...

    relPath, err := RenderExportPath(d.ExportPathTmpl, kind, req.Namespace, req.Name)

    // Git 연산을 직접 수행하지 않고 Queue에 Enqueue
	d.GitQueue.Enqueue(git.ExportJob{Path: relPath, Content: yamlBytes})
	return ctrl.Result{}, nil
}
```

이유 1. Go 언어와 client-go의 강력한 내장 큐

이유 2. 장애 복원력 (Pod가 죽어서 인메모리 큐가 날아간다면?)
Redis 같은 외부 스토리지를 쓰는 가장 큰 이유는 "서버가 죽었을 때 큐에 쌓인 데이터가 날아가는 것을 막기 위함(Persistence)"입니다.
하지만 쿠버네티스 오퍼레이터는 다릅니다. 만약 inventory-controller Pod가 OOM(Out of Memory)으로 죽어서 큐에 있던 Git Push 대기열이 날아갔다고 가정해 보겠습니다.
Pod가 다시 재시작(Restart)됩니다.
컨트롤러는 클러스터에 있는 모든 대상 리소스(App, Secret 등)를 처음부터 다시 쫙 긁어옵니다(List).
긁어온 리소스의 해시를 떠서 현재 Git에 있는 상태와 다시 비교합니다.
"어? 이 리소스는 Git에 백업이 안 되어 있네?" 하고 스스로 판단하여 알아서 다시 큐에 집어넣습니다.
즉, 쿠버네티스 API 서버 자체가 이미 거대한 상태 저장소(Database) 역할을 해주기 때문에, 컨트롤러는 언제든 재시작해도 누락된 작업을 스스로 복구(Self-healing)할 수 있습니다. 무거운 외부 브로커를 붙여 관리 포인트를 늘릴 이유가 전혀 없는 것이죠!
이러한 이유로 우리는 순수하게 컨트롤러 내부의 Go 로직만으로 가볍고 강력한 큐를 구현하게 됩니다.

1. Informer 패턴의 로컬 캐싱 (단 한 번만 긁어옵니다)
컨트롤러는 매번 API 서버에 쿼리를 날려 데이터를 가져오는 방식이 아닙니다. client-go의 Informer라는 영리한 메커니즘을 사용합니다.
Pod가 처음 뜰 때(재시작될 때) 단 한 번 전체 대상 리소스를 긁어오고(List), 이후부터는 API 서버와 연결된 스트림(Watch)을 통해 변경된 이벤트만 실시간으로 밀어 받습니다. 긁어온 전체 데이터는 컨트롤러의 '로컬 인메모리 캐시'에 저장되므로, 이후 로직에서 발생하는 수많은 읽기 작업은 네트워크를 전혀 타지 않아 지연 시간(Latency)이 0에 가깝습니다.
2. 페이지네이션(Chunking)을 통한 API 서버 보호
"그럼 재시작 시점에 단 한 번 긁어오는(List) 행위 자체가 API 서버나 etcd를 뻗게 만들지 않을까?" 하는 합리적인 의문이 남습니다.
이를 방지하기 위해 client-go는 내부적으로 **페이지네이션(Pagination)**을 수행합니다. 만약 대상 리소스가 10,000개라면, 무식하게 한 번에 10,000개를 달라고 요청하지 않습니다. 500개씩 잘라서(Chunk) 달라고 요청하며(Limit & Continue 토큰 활용), API 서버에 순간적인 병목(Spike)이 발생하지 않도록 부하를 스스로 조절합니다.
3. 현실적인 스케일(Scale)의 차이
Argo CD로 관리하는 Application이나 연동된 Secret(Cluster 정보 등)의 개수가 현실적으로 몇 개일지 가늠해 볼 필요가 있습니다.
보통 수백 개에서, 아주 거대한 마이크로서비스 조직이라 할지라도 수천 개 수준입니다. 쿠버네티스 API 서버와 저장소(etcd)는 수십만 개의 Pod와 수시로 변하는 이벤트를 초당 수천 건씩 처리하도록 설계된 괴물 같은 시스템입니다. 컨트롤러가 재시작될 때 수천 개의 CRD 텍스트를 한 번 읽어가는 것은 API 서버 입장에서는 말 그대로 '먼지' 수준의 부하에 불과합니다.
🚨 예외적으로 대비해야 할 시점
단, 관리하는 Argo CD 리소스가 5만 개 ~ 10만 개 이상을 넘어가는 초대형 글로벌 멀티 클러스터 환경이라면 이야기가 조금 달라집니다.
이때는 API 서버의 부하보다는, **inventory-controller Pod 자체가 수만 개의 리소스를 로컬 캐시에 들고 있기 위해 소모하는 메모리(RAM)**를 걱정해야 합니다. (이 시점이 오면 컨트롤러 Pod가 OOM(Out of Memory)으로 죽지 않도록 Memory Limit을 수 GB 단위로 넉넉하게 열어주기만 하면 됩니다.)
그 정도의 극단적인 규모가 아니라면, 재기동 시 발생하는 크롤링 지연이나 부하에 대한 걱정은 완전히 내려놓으셔도 좋습니다. 우리가 설계한 내부 큐(Queue)와 Informer 아키텍처가 모든 것을 안정적으로 처리해 줄 것입니다!
